Backtesting Platform: Research Spike

Three lenses before we commit to a feature list — what the proven engines look like inside (A), what a trustworthy result even requires (B), and how this becomes a business if it earns one (C).

Scope: Stocks only Stack: Python Personal tool → product-ready Date: 2026-07-17

Summary & Verdict

The quant backtesting space is old and well-trodden. Rebuilding from first principles means re-learning its hard lessons the expensive way — so the research spike's job is to extract proven patterns and known traps, not a feature checklist to copy.

Verdict

Build a modular, handler-style Python engine with a strict split of data / strategy / portfolio / broker / analytics, driven by config + data → results pure runs. Use a hybrid engine: vectorized indicators, bar-by-bar portfolio simulation with next-bar-close fills and first-class cost models. Ship no-look-ahead tests from day one. Build like a product, scope like a personal tool — keep the door open for live-trading, a hosted API, and a licensable data layer, but don't build any of those yet.

The viable businesses in this space are consistently engine (free/OSS) + data or hosting (paid) — not the calculator itself. Design the engine to be clean and open, and design the data layer to be a separable, licensable asset.

A · Architecture Patterns

What proven engines look like inside

Five engines cover the design space. Each teaches one distinct lesson; together they map the decisions we face.

EngineLanguage / ModelCore idea we can borrow
LEAN (QuantConnect) C# · handler-based, config environments Engine split into pluggable handlers (ISetupHandler, IDataFeed, ITransactionHandler, IRealtimeHandler, IResultHandler). Same handlers run backtest and live by swapping config "environments". Fee/slippage/brokerage are pluggable models.
backtrader Python · event-driven Cerebro orchestrates; Strategy.next() runs per bar; LineBuffer circular buffers hold time series; analyzers and observers are separated from strategy logic.
vectorbt Python · fully vectorized (Numba + Rust) Strategies are columns in arrays, not OOP classes. Fills ~1M orders in ~70–100 ms. Built for sweeping thousands of param sets in seconds.
nautilus_trader Rust core + Python control plane Compiled core for speed and determinism (fixed-point i64/i128 arithmetic). Research-to-live parity: identical execution semantics in sim and live.
zipline-reloaded / bt Python · Pipeline + portfolio tree Pipeline(columns, screen) → attach_pipeline → pipeline_output for factor/alpha research; bt does portfolio allocation as a tree of strategies. Clean alpha-vs-allocation split.

Patterns to adopt

  • Strict layer separation — data / strategy (alpha) / portfolio / broker (fill) / analytics as independent modules with clear interfaces.
  • Config-driven environments — a run is config + data → results. Design the backtest/live switch now (even if we never ship live) so the seam exists.
  • Pluggable data sources & cost models — the engine never knows where data came from or what the commission schedule is.
  • Hybrid engine — vectorized indicator/signal math (fast) + a bar-by-bar portfolio simulator (correct-by-construction, no look-ahead). Best of both for daily stocks.
  • Analytics as separate "analyzers" — metrics computed from results, never baked into the strategy.
  • Determinism by design — seeded RNG, snapshot-versioned data, fixed-point where rounding matters. Required for reproducible research and for any future live parity.

Patterns to avoid

Anti-patterns

Single global-state notebook — the #1 thing that kills a personal tool's path to product. Keep the engine importable.
Hardcoded data paths / personal config — bakes the tool to one machine.
Costs as an afterthought — unrealistic fills make every result suspect.
Same-bar-close fills — classic look-ahead (see lens B).

B · Tearsheet & Data Guardrails

What a trustworthy result requires

A backtester is only as good as its freedom from bias. This lens has two halves: the standard metrics a quant expects, and the data-quality traps that must become design guardrails.

The canonical tearsheet

MetricMeaningRough benchmark
CAGRAnnualized compounded return—
SharpeReturn per unit of total risk (Rp−Rf)/σ<0.5 poor · 0.5–1.2 ok · >1.2 good
SortinoReturn per unit of downside risk onlyhigher than Sharpe, same spirit
CalmarCAGR ÷ |Max Drawdown|<0.5 poor · >1.2 good
Max DrawdownLargest peak-to-trough decline<10% great · 10–20% ok · >30% poor
TurnoverFilled notional ÷ equity, per rebalance / annualwatch >1000%/yr (cost-prohibitive)
Exposure / OccupancyAvg invested fraction, time-in-market, open positionscontext for risk

Supplement with trade stats (win rate, profit factor, expectancy) and cost attribution. Red flags: Sharpe > 2.5 unexplained, drawdown recovering instantly, no down years in 5+ yrs, turnover > 1000%/yr. Tools: pyfolio, ml4trading (adds Deflated Sharpe Ratio for overfitting).

Data-quality traps → design guardrails

Look-ahead bias

Three forms: direct (tomorrow's close decides today's trade), data-revision (using revised macro vs initial release — use ALFRED vintages), knowledge (conditioning on info unavailable in real time). The pandas alignment trap: a signal using today's close to trade the same bar. Fix: .shift(1). Detect: the shift test — lag inputs one bar; a clean strategy shouldn't collapse.

Survivorship bias

Backtesting only currently-listed stocks silently drops bankrupt/delisted names — the biggest losers — inflating returns by 1–2%/yr (up to 2–4% for hedge-fund-style universes). Fix: point-in-time constituent data + include delisting returns. Most free sources lack this — design the data layer to accept it later.

Split / dividend adjustment

Prices must be split- and dividend-adjusted consistently, or returns are garbage. Make adjusted price the canonical stored series; keep raw available.

Overfitting / data snooping

Massive parameter sweeps invite false discovery. Report Deflated Sharpe Ratio and prefer walk-forward over a single in-sample fit. The engine's config-driven runs make walk-forward trivial.

Guardrails to ship from day one

① Buy-and-hold check — 100% single asset ≈ that asset's return minus costs. ② Flat check — no-signal strategy = zero P&L. ③ Leakage check — signal at t uses only data ≤ t. ④ Cost check — with-cost ≤ without-cost, never reverse. These make the platform trustworthy and demoable later.

C · Business-Model Patterns

How this becomes a business, if it earns one

You said: personal tool first, product if it works. The market signal is consistent across every player that did monetize.

PlayerModelLesson
QuantConnect / LEAN OSS engine + cloud IaaS Land-and-expand: free/OSS entry → paid compute nodes → data subscriptions → enterprise/on-prem licensing. ~400 TB data library + 40+ vendor dataset marketplace. Virtual currency (QCC) for consumption billing. Data is the moat.
vectorbt OSS core + paid Pro tier Free engine, paid features/scale. Lighter version of the same land-and-expand.
backtrader Pure OSS, no commercial layer Great internals study, dead-end for business — proves you need a paid layer beyond the engine.
Polygon / Databento / iVolatility Sell data, not the engine The calculator is commoditized; clean, point-in-time data is the recurring-revenue asset.
The pattern to remember

Viable businesses = engine (free/OSS) + data or hosting (paid). Don't plan to monetize the calculator. Monetize compute (run users' configs on big machines) and/or clean point-in-time data (a licensable asset). Our config-driven runs map naturally onto a "compute node" later; our pydantic contracts map onto a hosted API; our pluggable data layer maps onto a marketplace.

D · Open-Source Landscape & Roadmap

Is an OSS project a good idea? (multi-agent research, 2026-07-17)

Four parallel research agents mapped the open-source quant world — backtesting engines, research tooling, execution/live-trading, and the strategic question of whether to launch a new open-source project. Verdict up front: yes, but only if it is narrow and opinionated — not "yet another engine." The detailed findings follow.

Gaps the OSS engines leave open

① Stocks + live parity without cloud/vendor lock-in — LEAN is C#/cloud, nautilus is crypto-first, and the stocks-focused OSS tools (zipline-reloaded, bt, backtesting.py, vectorbt) are backtest-only. ② No lightweight, permissively-licensed, actively-maintained event engine — backtrader is dormant, backtesting.py is AGPL, PyAlgoTrade is dead. ③ Built-in corporate actions / point-in-time fundamentals are DIY. ④ First-class ML + parameter-stability (walk-forward + overfitting guards) not combined for equities. ⑤ Agent/LLM-native strategy authoring is only nascent.

Gaps: sim/live realism (most paper modes ignore slippage/queue/dividends), thin equities broker coverage, maintenance fragility (ib_insync archived 2024), and no turnkey retail EMS.

D4 · Strategy verdict: build net-new, narrow, trusted

"Yet another backtester" fails on bus factor + maintenance burden — most quant OSS dies when the solo maintainer burns out, and forking a massive engine (LEAN=C#, nautilus=Rust) saddles you with a contributor-hostile surface you can't sustain. But the graveyard is full of engines, not of coherent, trustworthy, easy-to-adopt research tools. So: build net-new, but wrap, don't rebuild — use a vectorized back-end for sweeps and an event-driven back-end for verification; your code owns the contracts, the no-look-ahead guarantees, the reporting, and the data interface.

The wedge: no-look-ahead by default, enforced by architecture

Make trust the headline, not features. Concretely: strictly point-in-time data access, a timestamp-ordered event loop, an automated look-ahead & overfitting audit (Deflated Sharpe, backtest-overfitting score) run on every result, and zero global mutable state. The engine should refuse to compile strategies that can leak. This is verifiable, demoable, and a genuine pain no incumbent markets as its headline.

Phased open-source roadmap

v0.1 · Personal tool (now)

Event-loop core + point-in-time data loader + one report. Prove the no-look-ahead guarantee holds. Not public.

v0.5 · Launch-worthy wedge (~3 mo)

Dead-simple pip install, 3 example strategies that fail loudly if edited to leak, and the automated overfitting audit. Launch trigger: a stranger runs a trustworthy backtest in 5 minutes. Lead with the look-ahead guarantee.

v1.0 · Community growth

Plugin data interface, GitHub Discussions, contributor guide, CI with the audit as a gate. Begin contributing broker/data adapters upstream to borrow goodwill.

v2.0 · Sustainability layer

Paid point-in-time data subscriptions + optional hosted compute (open-core, à la QuantConnect). Engine stays free/OSS; monetize the data + compute you already maintain.

Top risks

① Bus factor 1 → keep surface tiny; design for contributor onboarding from v0.5; plan a commercial entity. ② "Just another engine" apathy → market trust + 5-minute start, never feature lists. ③ Data licensing cost/legal → free/own-sourced PIT data first; paid data is v2. ④ Overfitting theater → the audit must be mandatory and honest. ⑤ Incumbent co-opting → guarantee + Pythonic API + trivial install is hard for C#/Rust incumbents to copy without rewriting.

Synthesis · Our Design

What we steal, what we defer

Translating A + B + C into a concrete shape for a stocks-only Python platform:

  • Module layout — data / strategy / portfolio / broker / analytics / config / tests, each with a clean interface (LEAN's handler idea, backtrader's analyzer separation).
  • Engine — hybrid: vectorized numpy/pandas indicators → bar-by-bar portfolio simulator → fills at next-bar close (configurable fill_lag), with pluggable commission + slippage models (costs first-class, per lens B).
  • Runs are pure — config (pydantic) + data → results. Enables parameter sweeps, walk-forward, and a future API/server with zero engine changes.
  • Data store — fetch (Yahoo/CSV) → normalize → save as Parquet locally, snapshot-versioned. Point-in-time-aware hook left open for later.
  • Determinism & tests — seeded RNG, no-look-ahead + correctness guardrails from day one (lens B). This is also what makes a future product credible.
  • Product seams (designed, not built) — config environments for live later, pydantic API contract, separable data layer, deterministic runtime. None built now.

Feature Shortlist

Build-now · keep-door-open · defer

Build now (v1 — the trustworthy vertical slice)

Data layer: CSV / Yahoo → Parquet Hybrid engine, daily bars 1–2 example strategies (momentum / SMA) Cost model (commission + slippage) Tearsheet (7 metrics + trade stats) No-look-ahead + correctness tests CLI / notebook entry point

Keep the door open (design for, don't build)

Config "environments" seam (live later) Pydantic API contract Pluggable data → marketplace hook Point-in-time data layer hook Deterministic seed / runtime Thin FastAPI server (later)

Defer (explicitly out of scope)

Web UI Auth / multi-tenancy Billing Strategy marketplace Live trading Minute / intraday bars Options / futures
Why this is safe

Building the vertical slice proves the whole stack. The "door-open" items cost almost nothing now (clean boundaries, config, tests) yet prevent a rewrite if you later productize. The "defer" items are pure YAGNI until there's a real reason.
